Voyager

Voyager is Janeway's design system. It draws inspiration from two systems we admire — GitHub Primer and the UK Government Design System — but Voyager is its own thing. Every component is hand-written for Janeway's editorial admin context; nothing is imported from either source.

What inspired Voyager

Janeway is an editorial admin tool. Editors and managers spend a lot of time scanning lists, reading metadata, and navigating between queues. That points us toward the kind of compact density and chrome we admire in GitHub Primer — thin top bars, left rails, hairline borders, monospaced identifiers. Editorial actions are also consequential: sending a decision letter, withdrawing a reviewer, requesting revisions. That points us toward the accessibility discipline we admire in the UK Government Design System — summary lists, inset text, yellow focus rings, unambiguous link text, and a strict form pattern.

We re-write rather than re-use — the lessons are theirs; the code is ours.

How it's organised

  • Foundations — the tokens, type scale, and layout primitives that everything else builds on.
  • Components — self-contained patterns. Each component lives in its own folder with co-located .html, .css, and .j2 macro. The CSS uses a single root class with native nesting; variants compose as additional flat classes.
  • Examples — three real Janeway pages built entirely from Voyager components: the editor dashboard, the manager index, and the peer-review screen.

How to use it

Add one stylesheet to your page and use the component HTML or call the Jinja2 macros:

Under review Article 
<link rel="stylesheet" href="/voyager/assets/css/index.css">

<span class="tag tag-green">Under review</span>
<span class="tag tag-grey">Article</span>
{% raw %}{% from "voyager/components/tag/tag.j2" import tag %}

{{ tag("Under review", kind="green") }}
{{ tag("Article", kind="grey") }}{% endraw %}

Architecture

Voyager follows the Memory Alpha CSS approach: ITCSS layer order (settings → reset → elements → components → utilities), flat component folders, single root class per component, native CSS nesting, composable variant classes (no BEM), two-tier tokens (raw + semantic). See assets/css/README.md for the full architecture document.

Hard rule: no inline styles, anywhere. If a one-off margin or width is needed, add a utility class.
