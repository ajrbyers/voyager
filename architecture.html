Architecture

Voyager is plain modern CSS — no preprocessor, no Tailwind, no framework. Native CSS nesting and custom properties only. The system follows an ITCSS-style layered structure with co-located components, two-tier tokens, and a small set of hard rules.

Inspiration

Voyager is inspired by two design systems we admire — GitHub Primer for chrome density and admin patterns, and the UK Government Design System for accessibility discipline and form conventions. Voyager owes its visual rhythm to Primer and its content semantics to GDS, but every component is hand-written for Janeway's editorial admin context. Nothing in Voyager is imported from either system — the lessons are theirs; the code is ours.

Layer order

The cascade flows in one direction. Each layer is more specific than the one before it. Utilities load last so utility classes win over component styles. 

settings  →  reset  →  elements  →  components  →  utilities
LayerFileContains
Settings settings.css Tokens — colour, type stacks, type scale, space scale, radii, layout. Two-tier: raw tokens (--colour-grey-9) plus semantic aliases (--colour-fg).
Reset reset.css Box-sizing, margin / padding zeroing, list-style stripping where appropriate.
Elements elements.css Typography and link styles set on element selectors only (body, h1, a, p, …). No classes.
Components components/<name>/<name>.css One namespaced root class per component, all declarations nested under it with native &. No BEM.
Utilities utilities.css Single-purpose helper classes (.muted, .mono, .visually-hidden, .stack, .cluster). Win over component styles.

Component file pattern

Each component lives in components/<name>/ with three co-located files:

  • <name>.html — the canonical markup, copy-pasteable
  • <name>.css — single root class, native nesting, no element selectors
  • <name>.j2 — illustrative Jinja2 macro for production use

The CSS opens with the root class and nests everything inside it with the native & selector. Variants are extra flat classes composed in the markup — not BEM modifiers. 

/* button.css — root class is .btn, everything nests under it.
   Variants compose as additional flat classes in the markup. */

.btn{
  font-family:var(--font-sans);
  background:var(--colour-surface-grey);
  …
}
.btn:hover{ … }

/* Variants — extra classes, not BEM modifiers */
.btn-primary{ background:var(--colour-success);color:var(--colour-white) }
.btn-warning{ background:var(--colour-error);   color:var(--colour-white) }

/* Sizes compose with any variant */
.btn-small { padding:4px 8px 3px;font-size:var(--fs-sm) }
.btn-large { padding:9px 14px 8px;font-size:var(--fs-lg) }

Composable variants vs BEM

A button is btn btn-primary btn-large, not btn btn--primary btn--large. The visual axes (kind, size, intent) compose as separate flat classes, so any combination is one class list: 

<button class="btn btn-primary">Save</button>
<button class="btn btn-primary btn-small">Save</button>
<button class="btn btn-warning btn-large">Delete</button>
<a class="btn btn-start" href="/start">Start review</a>

Tokens — two-tier system

All design tokens live on :root in settings.css. Two tiers:

  1. Raw tokens — the underlying value: --colour-grey-9: #1f2328
  2. Semantic aliases — point at raw tokens by role: --colour-fg: var(--colour-grey-9)

Components reference semantic tokens only. Pages and themes can override semantic aliases without touching raw tokens. This makes a future dark mode straightforward — alias remapping in a single [data-theme="dark"] block. 

:root {
  /* Raw */
  --colour-grey-7:  #1f2328;
  --colour-blue-5:  #0969da;
  --colour-green:   #00703c;

  /* Semantic aliases — components reference these */
  --colour-fg:      var(--colour-grey-7);
  --colour-accent:  var(--colour-blue-5);
  --colour-success: var(--colour-green);
}

Hard rules

Non-negotiable. The system relies on these to stay coherent at scale.

  1. No inline styles. Anywhere. If a one-off margin or width is needed, add a utility class.
  2. No style="…" attributes in component HTML or example pages. Verified with a repository-wide grep.
  3. Element selectors stay in elements.css. Don't add element selectors to component files — reach for a class.
  4. Components don't reference each other's classes. A component is self-contained. Composition happens in the markup, not cross-component CSS.
  5. Utilities are a last resort and load last. Prefer composing existing components.
  6. Accessibility is a quality bar. WCAG AA contrast, real focus rings (yellow, inspired by GOV.UK), semantic HTML, no fake ARIA, real <ol>/<ul> for lists, aria-current on active nav.
  7. Squared edges everywhere — no border-radius on rectangles. Inherently circular elements (avatars, dots, step badges, the password show/hide circle for keyboard focus) use border-radius: 50%.
  8. Use the .tag chip baseline for all chip-style elements. Don't invent a parallel chip — extend or compose.
  9. Inline SVG icons only — no unicode glyphs in component icon slots. See the icon component.
  10. Macros follow the wrapper + item pattern for any component that contains a repeating child (folder list, tabs, breadcrumbs, steps, action menu, error summary). Compose via {% raw %}{% call %}{% endraw %} in templates rather than passing data dictionaries.

Bundle

index.css is the single entry point. Pages link only to that file. It uses @import to pull in every layer file in order; authoring stays split. 

<link rel="stylesheet" href="/voyager/assets/css/index.css">

For production deployment, run a CSS concatenator (any modern bundler will flatten the @import chain) and ship a single minified voyager.min.css. Authoring stays in the split component files; the build flattens.

Provenance

The architecture is adapted from Memory Alpha, OLH's Pelican-based site theme. Memory Alpha pioneered the co-located component pattern with native CSS nesting and composable variant classes — Voyager carries those decisions forward, adds form discipline inspired by GOV.UK, chrome conventions inspired by Primer, and a stricter "no inline styles" stance.
