Progress bar

Thin 3-pixel strip pinned to the very top of the viewport that animates during async work. Pair with HTMX events (or any fetch lifecycle): start the bar on request, finish it on settle. Purely visual — never rely on it for status messaging.

Try it

One progress bar is mounted at the top of this page. Click the button below to run the standard start/finish lifecycle.

</button>

Markup

One element. Toggle the is-running class and set width from JavaScript.

The progress bar above pins to the top of the viewport, so it lives at body level rather than inline. Use the Try it button to run the lifecycle.

HTMX driver

Wire up HTMX request and settle events to animate the bar. The bar fills to 85% on request start (slow ease), then to 100% on settle and fades out.

JavaScript snippet for the standard HTMX wiring:

const bar = document.getElementById('progress-bar');

function start() {
  bar.style.transition = 'none';
  bar.style.width = '0%';
  bar.classList.add('is-running');
  requestAnimationFrame(() => requestAnimationFrame(() => {
    bar.style.transition = 'width 8s cubic-bezier(0.1, 0.05, 0, 1)';
    bar.style.width = '85%';
  }));
}

function done() {
  bar.style.transition = 'width 0.15s ease';
  bar.style.width = '100%';
  setTimeout(() => {
    bar.classList.remove('is-running');
    bar.style.width = '0%';
  }, 250);
}

document.body.addEventListener('htmx:beforeRequest', start);
document.body.addEventListener('htmx:afterSettle',  done);
document.body.addEventListener('htmx:responseError', done);

Notes

  • The bar is aria-hidden="true" — screen readers should pick up loading state from the live region of the changing content, not from the bar.
  • Don't show the bar for every HTMX request; gate it to the pane being loaded (compare evt.detail.target with the target element).
  • Use cubic-bezier(0.1, 0.05, 0, 1) for the slow first phase so the bar feels like it's making progress without ever finishing prematurely.
